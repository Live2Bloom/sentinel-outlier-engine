# Deployment and Infrastructure Guide
This guide outlines the procedures for deploying SentinelLog within a secure, simulated production environment.

## Infrastructure Strategy
SentinelLog is designed to operate as a lightweight monitoring node. In a standard deployment, the engine consumes logs generated by web gateways (Nginx/Apache) or application servers.

### Secure Log Acquisition
To maintain a security-first posture, logs should be retrieved from remote web servers via encrypted channels. 

#### 1. SSH Pipeline Setup
Establish a secure data pipeline to stream logs directly from a production server to the monitoring node:
```bash
ssh -i ~/.ssh/id_rsa user@web-server "tail -f /var/log/access.log" >> server_logs.csv
```

#### 2. Key Management
* **Authentication:** Use RSA 4096-bit keys for all automated connections.
* **Principle of Least Privilege:** The SSH user on the web server should have read-only access strictly to the log directory.
---
## Runtime Optimization
For maximum performance during high-traffic bursts, the C binary should be compiled with level 3 optimization:
```bash
gcc -O3 activity_tracker.c -o sentinel -lm
```

## Maintenance and Scalability
| Component | Maintenance Requirement |
| :--- | :--- |
| **Log Files** | Implement logrotate to prevent storage exhaustion. |
| **Memory** | Monitor RAM usage if scaling the table size beyond 100k users. |
| **Reset Cycle** | The engine utilizes a 10-minute windowing system. |
---

## Security Disclaimer
SentinelLog is a behavioral detection tool. It is intended to function as a component of a defense-in-depth architecture and should be deployed alongside managed IAM policies and firewalls.
